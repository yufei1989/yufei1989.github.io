RocketMq高并发场景应用

发表于 2019-06-29 | 更新于: 2019-07-03 | 分类于 IT技术
字数统计: 3.1k | 阅读时长 ≈ 12

概述

本文主要讲述rocketMq在高并发场景中能为系统提供怎样的并发能力，又是如果做到的。首先，我们来回顾下什么是高并发，在高并发下有哪些点会影响系统的性能。所谓高并发，引用一段架构师之路公众号究竟啥才是互联网架构高并发中的一段话来描述:

高并发（High Concurrency）是互联网分布式系统架构设计中必须考虑的因素之一，它通常是指，通过设计保证系统能够同时并行处理很多请求

但是很多请求没有明确数量上的定义，一般PV在千万级别以上的公司才会涉及到这个概念，如果某个系统的日PV在千万级别以上，那么该系统可能就是一个高并发系统。当前衡量指标常用的有响应时间，吞吐量，每秒响应请求数QPS，并发用户数。

  • PV:(page view)，即页面浏览量；用户每1次对网站中的每个网页访问均被记录1次。用户对同一页面的多次访问，访问量累计。

  • 响应时间：系统对请求做出响应的时间。例如系统处理一个HTTP请求需要200ms，这个200ms就是系统的响应时间。

  • 吞吐量：单位时间内处理的请求数量。

  • QPS(TPS)：每秒响应请求数。

  • 并发用户数：同时承载正常使用系统功能的用户数量。例如一个即时通讯系统，同时在线量一定程度上代表了系统的并发用户数。

如何提高并发能力

并发是指，某个时刻有多少个访问同时到，QPS是指秒钟响应的请求数量。那么这里就肯容易推算出一个公式：
QPS = 并发数 / 平均响应时间

系统并发量的计算方法

每台服务器每秒处理请求的数量 = ((80% * 总PV量)/(24小时 * 60分 * 60秒 * 40% )) / 服务器数量
其中关键的参数是80%、40%。表示一天中有80%的请求发生在一天的40%的时间内。24小时的40%是9.6小时，有80%的请求发生一天的9.6个小时当中（很适合互联网的应用，白天请求多，晚上请求少)。

提高并发

实际上如果在单机上，要求我们每秒钟处理请求必须达到 61.7 以上才行，而实际上我们当前系统的qps是 10。那么怎么解决？

加服务器

单台机器如果qps实在上不去，那团队的力量是无穷的，一台搞不定，就多台机器。这里就涉及到比如数据库端，db主从、读写分离；应用服务端的请求负载均衡。本文不做过多的分析。

以上的只是一些服务器配置方面的简单调优，这里不做过多讲解。除了服务器性能配置，如果要提升请求的并发能力，只能是通过我们业务代码的优化，下面我来分享下，在电商应用中交易链路中的很重要的一块，下单业务流程，如何提升高并发场景的优化，仅供参考。

交易链路下单优化

电商应用中，下单业务包含如下几个重要流程节点：

  • 下单商品是否有效
  • 商品库存的校验
  • 扣减库存
  • 生成订单
  • 生成商品销量数据

简化下单流程

以上的流程，在并发不高的情况下，基本是没有大的问题，但是如果在高并发场景下，如：下单的商品参与秒杀活动或营销活动，在流量暴增的情况下，查询库存与扣减库存还是这种方式，是否会有问题？

问题排查

我们知道，mysql在索引上更新数据时，该事物会获取该数据行的排他锁即行锁，其他事务就不能再获取该行的其他锁（排他锁或者共享锁），即一个事务在读取一个数据行的时候，其他事务不能对该数据行进行增删改查，这时会导致所有请求阻塞在db端，导致系统的并发性能很差

解决方案

  • 库存行锁优化
    • 扣减库存缓存化
    • 异步同步库存到数据库
    • 保证库存数据库与缓存库存最终一致性

根据方案，我们可以来选型所需要的技术，缓存可以使用redis,而异步同步缓存库存与db库存，这里由于RocketMq支持事物型消息，可以保证最终一致性，所以我们选用RocketMq消息方式与库存中心通信。

优化下单流程

通过redis缓存库存，释放来自DB压力，如：某个商品参加秒杀活动或营销活动时，同步库存数据到redis,下单时，直接扣减redis缓存，成功后流程继续往下流转，依靠redis高性能，这一步可以提升很高的性能，同时依靠RocketMq良好的水平扩展能力以及高性能，与库存中心解耦，异步扣减DB的库存。这里有个疑问，为何使用RocketMq的事务型消息？

我们思考下，如果本地事物A成功后，系统挂了，没有来得及发送消息，DB端库存数据与redis缓存库存数据出现不一致的状况了，那有什么方案解决此种情况么。要保证可靠消息有两种方法。

业务与消息耦合的方式

redis在完成扣库存的同时，同时记录消息数据，这个消息数据与业务数据保存在同一数据库实例里（消息记录表表名为message）。

上述事务能保证只要本地事物成功，消息一定能保存下来。

当上述事务提交成功后，我们通过实时消息服务将此消息通知库存中心，库存中心处理成功后发送回复成功消息，订单中心收到回复后删除该条消息数据。

业务与消息解耦方式

上述保存消息的方式使得消息数据和业务数据紧耦合在一起，从架构上看不够优雅，而且容易诱发其他问题。为了解耦，RocketMq从4.3.0之后版本支持分布式事务消息。

RocketMQ第一阶段producer发送Prepared消息时，会拿到消息的地址，第二阶段执行本地事物，第三阶段通过第一阶段拿到的地址去访问消息，并修改消息的状态。

细心的你可能又发现问题了，如果确认消息发送失败了怎么办？RocketMQ会定期扫描消息集群中的事物消息，如果发现了Prepared消息，它会向消息发送端(生产者)确认，库存到底是减了还是没减呢？如果减了是回滚还是继续发送确认消息呢？

RocketMQ会根据发送端设置的策略来决定是回滚还是继续发送确认消息。这样就保证了消息发送与本地事务同时成功或同时失败。

客户端发送事务消息的部分代码

消费事务消息
这个时候就会出现消费失败和消费超时两个问题，解决超时问题的思路就是一直重试，直到消费端消费消息成功。这样基本上可以解决消费端超时问题，但是如果消费失败怎么办？阿里提供给我们的解决方法是：人工解决。大家可以考虑一下，按照事务的流程，因为某种原因扣减库存失败，那么需要回滚整个流程。如果消息系统要实现这个回滚流程的话，系统复杂度将大大提升，且很容易出现Bug，估计出现Bug的概率会比消费失败的概率大很多。这也是RocketMQ目前暂时没有解决这个问题的原因，在设计实现消息系统时，我们需要衡量是否值得花这么大的代价来解决这样一个出现概率非常小的问题，这也是大家在解决疑难问题时需要多多思考的地方。

总结

方案没有最好只有更好，大家在设计方案的时候，要充分考到当前业务，如过当前业务并发场景不高，我们也就没必要去引入一堆中间件，反而得不偿失。

本篇主要是提供一些在并发场景下，能优化的地方，经验有限，如有错误地方，欢迎一起讨论！
